# Architecture

Our Recipes uses a Next.js App Router application with a deliberately narrow vertical-slice structure:

- `src/app`: presentation routes and versioned HTTP endpoints only.
- `src/components`: accessible client interactions and visual surfaces.
- `src/lib/domain`: validation and domain input types.
- `src/lib/services`: household and profile use cases, independent of HTTP.
- `src/lib/storage`: server-only durable media storage and image transformation boundary.
- `src/lib/db`: Drizzle table definitions, SQLite connection lifecycle, and migrations.
- `src/lib/actor-context.ts`: the non-auth profile selection seam used by later audit/history features.

The current runtime is Node 24, Next 16, React 19, strict TypeScript, Tailwind 4, Drizzle ORM, `better-sqlite3`, Sharp, and local PDF.js text extraction. `DATA_DIR` defaults to `./data`; SQLite is stored at `DATABASE_URL`, which defaults to `DATA_DIR/our-recipes.db`. Recipe images are stored below `DATA_DIR/uploads`, while normalized document imports are stored below `DATA_DIR/generated/imports`; neither tree is in the application web root. The database uses WAL mode and foreign keys. `pnpm db:migrate` applies the committed Drizzle migration before a deployment, while the application also ensures current migrations before serving state.

The browser never opens SQLite, filesystem paths, or an AI provider. API route handlers validate input with Zod, check an exact trusted mutation origin, call services, and return versioned JSON. No CORS headers are enabled by default.

Profiles are a convenience layer, not identity or authorization. The selected profile is held in an HttpOnly, same-site, HMAC-signed cookie. `ActorContext` exposes the selected profile ID without making a security claim. A future authenticated deployment can replace that seam without changing recipe-domain services.

Household organization is server-backed: editable profiles carry cooking preferences and may be archived only when another active profile remains; profile archival never deletes attributed recipe/history rows. Tags are a canonical local catalog with optional colors and usage counts. Their rename, merge, and delete services update recipe associations and affected FTS documents atomically; browser tag autocomplete is advisory, while the server normalizes all saved names.

Curated collections use their own server-only service and additive SQLite tables rather than adding organization state to recipe content. Collection membership is ordered, many-to-many, and transactional; a collection edit does not create a recipe revision. Covers reuse only an already-normalized local recipe image from a current member recipe, so collections do not widen the upload or remote-image trust boundary.

Planning is likewise a server-side service workflow rather than browser state. A meal entry identifies either a saved recipe or a bounded free-form title across breakfast, lunch, dinner, and snack; copying a week makes independent dated entries and deterministic ICS export reads only the selected range. Shopping-list generation uses only structured recipe entries, preserves source-recipe IDs, and never invents ingredients for free-form meals. Household store aisles are a separately ordered catalog. The list UI derives accessible groups from that saved order plus an explicit unassigned group, while item edits/reorder and aisle removal stay durable through the existing versioned API boundary.

The recipe package stores a normalized current graph with ordered ingredient/instruction/equipment rows, rich metadata/lifecycle fields, user-entered nutrition values, and immutable JSON revisions. SQLite FTS5 indexes the current title, summary, ingredients, tags, category, and cuisine; create/edit transactions refresh it with the structured rows and revision record. Every shared replacement or lifecycle write requires the current revision and returns a conflict rather than silently replacing a newer household edit. Restore reads one existing snapshot through the same bounded recipe schema and appends it through the normal update path, preserving every older revision. Profile ratings and kitchen notes are held in a separate `(profile_id, recipe_id)` table and never enter a shared snapshot or exporter. The selected profile’s rating/favorite state is read only into its library/detail projection; highest-rated order never aggregates ratings across profiles. The browser editor saves only local recovery data in browser storage, warns on browser unload while dirty, and uses keyboard-reachable move controls for sections, ingredients, and steps. Local image routes accept only bounded JPEG/PNG/WebP bytes, call the server-only Sharp pipeline, and expose processed opaque-key media through a recipe-scoped endpoint. Markdown export is a pure local read transform; nutrition remains user-entered data with no enrichment service.

Document import follows the same server-only storage boundary. The `/imports` API derives PDF/image type from bytes and accepts either one PDF or an ordered set of one to four JPEG/PNG/WebP scans inside a fixed aggregate byte limit. A browser state machine records selection before asynchronous work, accepts both `input` and `change` events for mobile Safari, and exposes explicit preparing, ready, error, and submitting states. HEIC/HEIF preparation tries native browser image decoding first and falls back to the bundled browser-only WASM decoder, with a fixed client timeout. Both paths create an in-memory JPEG before multipart construction; the raw file never crosses this boundary and still fails byte validation if posted directly. PDF.js parses only supplied bytes with fixed page/text limits; each accepted scan is independently pixel-bounded, orientation-normalized, metadata-stripped, and stored as an opaque local WebP. `import_artifacts` records each artifact's opaque ID, source name/hash, media type, and order. With the UI's default vision choice and no transcription, the server creates a local pending review operation, then the same explicit button action sends only its normalized scans to the credential-gated OpenAI vision provider. A returned candidate remains editable and review-only. The bounded local Tesseract worker remains a printed/legible-text fallback for compatible callers; textless PDFs require manual transcription. Confirmation alone calls the recipe service.

Schema.org portability is deliberately separate from the file-import persistence path. `jsonld-service` receives only bounded pasted request text, parses JSON with a shallow root/array/`@graph` traversal, recognizes Schema.org `Recipe` types, and maps a selected candidate into the existing recipe input schema. The source text and candidate are never persisted; the browser must select a candidate and submit the editable recipe to a confirmation route before the recipe service writes anything. The export route maps an existing current recipe deterministically to a `Recipe` JSON-LD document and omits local image URLs, actor attribution, notes, tips, storage identifiers, and revision data. It performs no outbound request.

Public URL capture preserves that separation and the existing `safe-url-fetcher` controls. After DNS/address validation, redirect revalidation, text/HTML type checking, timeout, and response-byte bounds, `capture-service` parses the returned HTML with server-only Cheerio. It does not render the page, execute scripts, or request subresources. Candidate selection prefers embedded Schema.org JSON-LD (including multiple `@graph` nodes), then Microdata; Open Graph metadata and bounded readable text only enrich an explicitly warned fallback draft. The browser resubmits the selected index with the URL, so no page HTML or candidate state is stored between review requests.

The read-only PWA is intentionally implemented without a framework cache plugin. `src/app/manifest.ts` provides App Router manifest metadata, `PwaRegistration` registers `/sw.js` only in secure contexts, and the worker maintains a versioned Cache API store. It caches successful same-origin recipe-library/detail navigation responses, recipe GET APIs, normalized local image reads, and immutable Next static assets. It never intercepts a write for caching or replay, and an uncached/offline navigation falls back to the static `/offline` page.

Backup/recovery is a server-only service boundary. It snapshots SQLite through the database driver's backup API, copies only regular files from the durable `uploads` and `generated` trees, records checksums in a versioned manifest, and writes a server-created `.tar.gz` bundle below `DATA_DIR/backups`. Preview validates the archive in an isolated staging directory before restore is even offered. Restore accepts only a listed local bundle, validates paths/checksums/database integrity again, creates a safety backup, swaps data roots with rollback protection, then reopens and migrates SQLite. The process-local scheduler is intentionally bounded to local storage and retention; remote backup and container orchestration remain later operational work.

Portable recipe export is a separate server-only, output-only boundary. It reads the current recipe graph and its referenced normalized media through service APIs, then stages deterministic Schema.org JSON-LD and WebP copies under fixed archive-relative names. The resulting `tar.gz` contains `manifest.json`, `recipes/00001.jsonld`-style documents, and content-hash-addressed `images/*.webp` files; the manifest maps documents to images and records SHA-256/byte-size evidence. It is deliberately not a backup or restore format: no SQLite, profiles, preferences, revisions, source storage keys, imports, or configuration are staged. The stream is deleted after download, and no code path accepts this archive as input.

AI integration is a server-only provider boundary. `OpenAiProvider` uses the official SDK's Responses Structured Outputs API with an all-required provider schema followed by the application validator; it accepts bounded pasted text or existing normalized scan artifacts only. `.api_keys` is a development-only convenience path, while production reads only `OPENAI_API_KEY`; Docker excludes the file. `/api/v1/ai/status` reveals only configured/unconfigured state. Every review/generation request requires trusted origin, active profile, an explicit confirmation action, rate-limit capacity, and an audit row without raw source/provider URLs/secrets. Generated image bytes pass through the existing local image pipeline. Deterministic doubles cover this behavior; no test makes a paid network request.

Docker uses standalone Next output in a Node 24 Debian-slim runner. Its entrypoint initializes the dedicated `/data` mount as root and immediately drops to UID/GID `1001`; the application binds port `3000`, keeps all mutable state below `/data`, and runs a separate lock-protected migration step before the server. Compose can pass `OPENAI_API_KEY` at runtime only; the image never receives `.api_keys`. The local smoke script builds, health-checks, performs first-run setup, recreates a temporary bind-mounted container, and proves the setup persists. Unraid host configuration remains an operator-specific release check.

# API conventions

Food discovery uses profile-scoped, exact-origin POST actions because requests consume shared upstream quota and populate a transient server cache. The stable routes and errors are documented in `openapi.yaml`; provider behavior is described in [food-data-integrations.md](food-data-integrations.md). Raw upstream responses and credentials are never response fields.

Application routes are under `/api/v1`; the private Homepage integration is under `/api/integrations/homepage/v1`. Responses use JSON, all write endpoints require an exact same/trusted `Origin`, and failures use:

```json
{ "error": { "code": "machine_readable_code", "message": "Safe human explanation" } }
```

The API is intentionally not an authentication boundary. Do not expose it to the public internet without adding real authentication and a revised threat model.

Pantry reads are household-wide. Pantry writes require the same exact trusted-origin check as every other mutation plus a valid signed active-profile selection for attribution; profiles remain a convenience and are not authorization. Canonical products describe what an item is, while batches represent physically distinct stock with their own quantity, location, dates, state, and optimistic version. Exact quantity conversions are dimension-safe, approximate quantities are never silently treated as exact, and first-expiring-first consumption operates only across compatible exact batches. Every stock mutation appends an actor-attributed event containing bounded before/after state; undo reverses the newest eligible event by appending another event rather than deleting history.

The Pantry summary query supports active and inactive lifecycle views plus text, location, category, status, expiry-state, and sort filters. Split and combine are one atomic transaction: both paired batches receive immutable linked events, stale peer versions reject the operation, combine refuses incompatible metadata that would erase storage or expiry facts, and undo compensates both sides together.

Manual Pantry mappings attach to stable recipe ingredient IDs and never rewrite ingredient wording or append a recipe revision. Recipe availability scales an exact ingredient quantity from the recipe yield to the requested servings. Ingredients mapped to the same product draw sequentially, in recipe order, from one local calculation pool; physical stock is not changed. Its `ready`, `partial`, and `unknown` states explain the exact required, compatible available, and known shortage quantities; missing mappings, missing quantities or units, incompatible units, and approximate stock remain explicit unknowns. Projected demand aggregates all recipe-linked meals in the requested range and counts each compatible exact batch once. Each line is `covered`, `shortage`, or `uncertain`; an uncertain line has a null `shortageQuantity` and a non-null `uncertaintyReason` because approximate or incompatible stock could change the conclusion. Both range values must be real `YYYY-MM-DD` calendar dates. The forecast does not reserve, consume, version, or otherwise mutate Pantry stock.

The Homepage summary is the exception: it is a server-to-server read endpoint requiring `Authorization: Bearer <token>` matching the server-only `HOMEPAGE_INTEGRATION_TOKEN`. Keep the token out of browser code and logs, use a unique high-entropy value, and call the endpoint only over the trusted internal network (for example `http://bord:3000`). It returns at most five unchecked items from the newest grocery list in their saved order. The next eligible recipe-linked meal is evaluated in the setup profile's timezone; date-only meal slots are represented as breakfast 08:00, lunch 12:00, snack 15:00, and dinner 18:00 local time. Public recipe and image links always use `https://recipes.tower`.

The AI status endpoint has no enable/configure action and never returns a key. Production reads `OPENAI_API_KEY` only from the server environment; local development may use the ignored root `.api_keys` convenience file. `POST /ai/reviews` requires `confirm: true` plus either 20–30,000 characters of text or an existing image-import ID, and accepts an optional `improve` flag that asks the model to repair and enrich instructions and metadata while keeping the source ingredients. `POST /recipes/{recipeId}/improve` requires `confirm: true`, the current revision, and the current bounded editor recipe; it returns a review-only candidate and server-enforces the submitted ingredient groups even if the provider suggests changes. Image generation and nutrition estimation remain explicit actions. Every AI mutation requires a trusted origin and selected-or-default household profile, enforces a per-profile process rate limit, and writes a no-raw-content audit. No review or improvement endpoint auto-saves a candidate or returns provider URLs/secrets.

Recipe reads and writes include ordered ingredient groups, instruction sections, equipment, original-author/source/cooking-method metadata, optional user-entered nutrition values, current creator/editor attribution, and revision metadata. The editor sends `expectedRevision` on every shared replacement, lifecycle change, or restore; a stale value returns `409 recipe_revision_conflict` rather than overwriting a newer household edit. Restoring a saved revision validates its stored bounded recipe shape and appends a new revision—it never mutates the old snapshot. Profile ratings/notes are separate and do not increment that revision. Image-card reactions use the existing private rating field: dislike is 1, like is 3, and staple is 5, leaving 2 and 4 available for future reactions. Scores 3 and 5 also retain compatibility with the personal-favorites filter. Recipe-list reads default to active cards and 24 cards per page. Their bounded query parameters are `q`, `creator`, `tag`, `collection`, `category`, `cuisine`, `maxTotalMinutes`, `favorite=true`, `cooked=true`, `status`, `sort`, and `page`; `sort=highest-rated` uses only the selected profile’s rating and falls back to recency without one.

Tags are household-managed, lower-cased labels. Rename, merge, and deletion update recipe associations in a database transaction and refresh full-text search; deletion removes the tag association rather than a recipe. Profiles remain attribution/preferences only and are never an authorization boundary.

The portable full-recipe export is a download-only, no-store `application/gzip` response. It requires a signed active-profile selection and rejects an explicit untrusted `Origin`; a normal local browser download has no `Origin` header. The service creates a private temporary staging directory, writes only generated JSON-LD plus regular normalized WebP copies, records a format/version/snapshot timestamp, stable archive paths, byte lengths, and SHA-256 checksums in `manifest.json`, then streams the resulting tarball and deletes the staging/archive files. It accepts no archive bytes, files, paths, or restore request. Profile names/IDs, ratings/notes, revisions, import/OCR records, source storage keys, and configuration are excluded. The export covers all current recipe records; lifecycle and other application-only state remain recoverable through the separate local backup format.

Collections are household-managed named shelves. Recipe membership is ordered and many-to-many; replacing a shelf’s membership never changes the recipe graph or its revisions. A cover may refer only to an existing normalized photo of a recipe already in that collection. Removing a collection removes only its associations, and removing a covered recipe/photo clears the cover rather than retaining a broken reference.

Recipe-photo upload uses `multipart/form-data` fields named `image` and optional `altText`. It ignores the client MIME type and filename, rejects non-JPEG/PNG/WebP signatures, and returns `413` for photos above 10 MB. The response never exposes the durable storage key.

Document import accepts repeated `files` multipart fields (with legacy single `file` compatibility), optional `transcription`, and an optional `autoOpenAiVision=true` UI hint. It permits exactly one PDF or one to four JPEG/PNG/WebP scans, derives every accepted type from bytes, and caps all submitted source bytes at 15 MB. Browser HEIC/HEIF conversion replaces original files with JPEG before this request; optional bounded `clientConversions` metadata explains that provenance, but raw HEIC/HEIF remains rejected by the server. PDFs remain limited to 12 pages/100,000 extracted characters. Each scan is metadata-stripped and stored as its own opaque normalized WebP artifact with source hash/name/order. For an image scan with no transcription and this hint, the endpoint creates a local `openai-vision-pending` review operation; the UI then makes the separately trusted-origin, credential-gated `vision-extraction` request. No provider call occurs from file selection or the import endpoint itself. The candidate remains editable and confirmation is still the only recipe write. Without the hint, clear English scan sets may use the pinned local Tesseract model; blank, low-confidence, timeout, or unavailable local OCR returns `422 text_required`, while textless PDFs always need transcription. The route is rate-limited to six starts per active profile per ten-minute process window; `422 rate_limited` asks the caller to wait. A confirmed operation returns `409` on replay.

JSON-LD portability uses JSON request field `source` (paste text, maximum 1 MB) and optional `candidateIndex`. The first request returns only bounded Schema.org `Recipe` candidates from a root object, array, or `@graph`; a second request with the selected index returns an editable draft. No request follows a URL, reads a file, accepts an archive, or stores the pasted source. `POST /jsonld-drafts/confirm` accepts the ordinary bounded recipe shape after review. Export returns `application/ld+json` with the portable title, summary, yield, ISO 8601 prep/cook/total durations, category/cuisine/tags, source URL, structured ingredients, and structured steps; it omits actor data, household notes, tips, local media, and internal storage information.

Markdown export is a local, deterministic text rendering of the current shared recipe card, including optional author/source, cooking method, equipment, and nutrition values exactly as entered. It deliberately omits profile-specific ratings and notes. Nutrition is a bounded user input field only; neither route nor service performs a lookup, estimate, or provider call.

For `kind: "url"`, `POST /capture-drafts` keeps the existing safe server-side fetch controls and never requests browser-side page access. It returns `candidates` for bounded embedded Schema.org JSON-LD Recipe nodes (including `@graph`/multiple nodes), or Microdata Recipes when JSON-LD is absent. Resubmit the same URL with a displayed `candidateIndex` to receive an editable `draft`. If no structured Recipe candidate exists, an explicitly warned Open Graph/text draft remains review-only. HTML, extracted text, and candidate data are not persisted by this endpoint; image URLs are not downloaded.

Meal-plan POST input requires an ISO local date, a named meal slot, 1–100 servings, a bounded note, and exactly one useful source: a current recipe ID or a bounded free-form title. `POST /meal-plan/duplicate` accepts source/destination week starts and copies only valid entries. `GET /meal-plan/export` requires `start`/`end` ISO dates and returns `text/calendar` with attachment, `no-store`, and `nosniff` headers; it performs no remote calendar integration. Shopping-list generation intentionally skips free-form meals. Its output remains an independent editable list even when a source week later changes.

Shopping aisle names are 1–80 trimmed characters and unique for the household. Reorder requests must name the complete current aisle set, preventing a stale client from silently dropping an aisle. Item writes accept only a current aisle ID or an empty assignment. The list read returns aisles in saved position order; clients should render those groups followed by **Unassigned**. Deleting an aisle never deletes a list item.

Backup restore never accepts an uploaded archive. It operates only on a backup already listed by the server, validates its archive path/type/manifest/checksums/database integrity in isolated staging, and requires an active profile plus a trusted mutation origin for create/restore actions.

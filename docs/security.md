# Security boundary

Food providers are read-only and server-mediated. USDA keys stay in the server environment; provider calls, cache writes, and imports require an active signed actor and exact trusted origin. Logs and diagnostics exclude keys, raw queries/barcodes, headers, and upstream payloads. Camera access is limited to `camera=(self)` and trusted HTTPS.

This app is designed for a trusted private household network, not the public internet. Profiles are explicitly not accounts, passwords, or authorization. The UI shows this warning during setup and after setup.

Current controls:

- SQLite, migrations, filesystem paths, and OpenAI credentials are server-only.
- POST/PATCH endpoints require an exact trusted `Origin`; CORS is not enabled by default.
- The active-profile selection is a signed, HttpOnly, `SameSite=Lax` cookie; its secure flag is enabled in production.
- Production refuses a missing or short `COOKIE_SECRET`.
- Baseline CSP, anti-framing, nosniff, referrer, permissions, COOP, and CORP headers are applied application-wide.
- Input is Zod-validated at the HTTP boundary. OpenAI provider modules are marked server-only; `OPENAI_API_KEY` takes precedence, while ignored `.api_keys` may be read only in local development and is excluded from Docker.
- AI readiness reports only configured/unconfigured state. OpenAI review and image requests require trusted origin, active profile, literal confirmation, bounded source input, a four-action-per-profile ten-minute process limit, strict Structured Outputs, and an audit row containing only source digest/label, operation/model/status/timestamps, and local IDs. Selecting a scan and creating its local import operation never contacts the provider; a paid vision request begins only after the user presses the expressly labeled OpenAI review action. Raw source, provider URLs, and credentials are neither audited nor returned. Tests use deterministic doubles; no live paid request is made here.
- Production dependency releases are installed only through frozen pnpm metadata. The
  workspace narrowly resolves Next's vulnerable declared PostCSS child to pinned
  `postcss@8.5.17`; `pnpm audit --prod --audit-level=moderate` is a release-maintenance
  gate and was clean on 2026-07-13. See decision 0024 before changing that override.
- Public URL capture is server-only and draft-only: it rejects credentials, private/reserved resolved addresses, excess redirects, non-text content, large responses, and slow responses. Redirect targets are rechecked; normal tests use mocked DNS/fetch responses only.
- URL Recipe extraction receives only the already bounded HTML response. A server-only parser traverses static markup without executing page scripts, loading images, or following metadata URLs; it considers embedded Schema.org JSON-LD Recipe nodes, then Microdata, and otherwise returns an explicitly warned Open Graph/text review fallback. Candidate state and page HTML are never persisted, and a browser must explicitly select a candidate before review.
- Recipe-photo upload accepts only JPEG, PNG, and WebP signatures below 10 MB. HEIC/HEIF preparation tries native browser decoding and then a bundled browser-only decoder, has fixed pixel/output limits and a 30-second timeout, and produces a metadata-free JPEG in browser memory before multipart construction; raw HEIC/HEIF still fails the server's byte gate. Sharp decodes accepted bytes with a pixel limit, rejects excessive dimensions and multi-frame images, honors orientation, strips source metadata, bounds output to 1,600 pixels, and writes a normalized WebP using an opaque generated key below `DATA_DIR/uploads`.
- Recipe-photo reads are recipe-scoped and always serve `image/webp` with `nosniff`; neither user file names nor user paths reach the filesystem.
- Document imports accept one PDF or one to four JPEG/PNG/WebP sources with a combined 15 MB limit after byte-signature validation. Browser HEIC/HEIF conversion happens before this boundary, and a bounded conversion-label map only records that the original was not uploaded. Mobile selection handling records the original `File` objects immediately, supports Safari's input event behavior, prevents duplicate conversion attempts, and keeps every failure recoverable without sending bytes. PDFs are parsed only from supplied bytes with local PDF.js, have a 12-page and 100,000-character limit, and reject malformed/password-protected input. Each scan is decoded with Sharp pixel/frame limits, orientation-normalized, metadata-stripped, and stored as an opaque local WebP artifact. A filename is reduced to a safe display label and never used as a path; artifact reads require an import-scoped opaque ID, add `nosniff`, are private/no-store, and force PDF download rather than inline execution.
- Import creation requires an active profile and trusted origin and is bounded to six starts per profile per ten-minute process window. A no-transcription scan may create only a local `openai-vision-pending` operation; its separately confirmed vision review then uses the existing bounded, audited provider route. Clear English scan sets without that opt-in may use one locally packaged Tesseract WASM worker at a time (two bounded waiters); it resolves only a validated package filesystem path with cache writes disabled, accepts only the existing normalized image buffers, has a 45-second deadline, and returns a manual-transcription fallback for blank/low-confidence/timeout/error output. It neither accepts a remote URL nor uses the default CDN model path, telemetry, an OpenAI credential, or provider request. Textless PDFs remain manual. Every source hash/name/order, extraction method/text, local model/version/confidence metadata, warning, and confirmation actor are retained as local provenance for review. The OCR assist is not handwriting validation.
- JSON-LD portability is paste-only and capped at 1 MB before parsing. The parser accepts only JSON root objects/arrays and bounded root/`@graph` Recipe discovery; it never treats text as a URL, opens files, unpacks archives, or traverses arbitrary object graphs. Candidate discovery and review drafts are request-local rather than persisted. Trusted-origin, active-profile confirmation validates the ordinary recipe schema before any write; the export is a deterministic local transform with `nosniff` and no household-private fields.
- Portable full-recipe export is server-generated output only: an active profile is required, an explicit `Origin` must be trusted, and the gzip response is `no-store` with `nosniff`. It accepts no bytes, file names, archive paths, or restore input. Process-private staging receives only generated JSON-LD and copied regular normalized WebP files; source links/symlinks, missing media, JSON-LD/media files over 10 MB, a manifest over 16 MB, more than 10,000 recipes or 50,000 distinct images, and more than 1 GB uncompressed output are rejected. The manifest exposes only stable archive-relative paths, lengths, and SHA-256 checksums; it excludes profile identity, profile-scoped preferences, revisions, import/OCR data, storage keys, and configuration. Staging/archive files are removed after a completed, failed, or cancelled stream.
- Rich recipe metadata and equipment are bounded shared inputs. Imported/manual legacy Nutrition values remain labelled source evidence; normalized, versioned ingredient calculations are authoritative across Recipes, Planner, and Nutrition. Optional AI estimation is an explicit configured provider action and remains unverified evidence until normalized sources exist. Per-profile ratings and notes use the validated active-profile ID and are excluded from shared revision snapshots and exports, but profiles are not a confidentiality boundary.
- Recipe snapshots are internal bounded JSON produced by the ordinary recipe validator. A restore route requires trusted origin, a valid active profile, a positive source revision, and the caller’s expected current revision; it validates the saved snapshot again and writes a new revision rather than replacing historical data. The library projection exposes only the selected profile’s favorite/rating values, not another profile’s preference row.
- Meal-plan writes accept only a valid local date, one of breakfast/lunch/dinner/snack, bounded servings/note, and either a current recipe ID or a bounded free-form title. Calendar export is a deterministic local `text/calendar` response with `nosniff`; it does not contact a calendar provider. Shopping rows and aisle names/order are bounded and require a trusted origin; deleting an aisle unassigns rows rather than deleting household food data.
- The PWA cache is constrained to successful same-origin `GET` responses for already-viewed recipe reads, local images, the manifest, and static assets. It never caches or replays POST/PUT/PATCH/DELETE requests, does not queue background sync, and returns a static offline page for uncached navigation rather than simulating a write.
- Backup export uses a SQLite snapshot plus only regular local media files and safe display metadata; it excludes environment files and secrets. Restore accepts only a currently listed server-created bundle, validates archive entry paths/types/size limits in isolated staging, checks every manifest checksum and SQLite integrity, requires exact `RESTORE` confirmation, and creates a safety backup before any data-root replacement.
- The Docker runner is non-root and keeps mutable state only below `/data`. `COOKIE_SECRET` and trusted origins are runtime environment variables, never image layers or backup exports. The checked-in container configuration remains LAN-oriented and must not be exposed publicly without a real authentication/reverse-proxy design.

Before adding arbitrary archive uploads, bundled handwriting OCR, remote-image imports, nutrition providers, or remote backup, add the corresponding size/type limits, audit/retention rules, mock-based tests, and any required human permission. A live OpenAI request additionally requires explicit paid-call approval.

Pantry grocery and cooking mutations retain the exact trusted-origin and signed ActorContext gates. Projected uncertainty is never converted to a numeric shortage, and obsolete manual rows cannot expose stale generated sources or numeric claims. Shopping edits and their manual-override flags commit together. Purchase intake requires a bounded explicit operation key plus an active product and location; the key is stored before dispatch and retained across rejection, unknown outcome, and concurrent retry. Only confirmed success rotates it for a distinct later operation. Cook completion rejects absent literal confirmation, applies every FEFO update and event in one SQLite transaction, and attributes events to the active profile with recipe, planned-meal, and cook-session links. Undo is append-only and conflict-aware: any newer batch event, optimistic-version mismatch, repeated undo, or linked leftover blocks restoration, and inventory compensation cannot commit without the cook-plan state transition.

Grocery controls accept only the closed covered/ignore/inaccurate/reset/extra action set. Ignore means ignore Pantry stock for that row, not ignore the purchase need. Inaccurate inventory forces a nonnumeric review result. The browser never supplies actor identity, and merely checking or opening an item never creates stock; rich batch intake requires a separate confirmation.

Interactive Pantry editing does not weaken these gates. Split and combine require optimistic versions for every affected batch, accept only exact compatible units, reject lossy metadata merges, and write or undo both sides in one transaction. Household profiles remain attribution context, not authorization.

Meal-status writes require the existing trusted-origin check and signed ActorContext profile attribution. Profiles remain a household convenience and are not authorization. Recipe availability and planner demand are server-side read projections; they do not reserve, consume, or edit inventory. Recorded expiry context is presented as planning evidence only and never as a food-safety conclusion.

Nutrition mutations use the same trusted-origin gate and signed `ActorContext`. The server resolves an active household profile, validates its exact linked compatibility principal, and writes both identifiers atomically; request bodies and query strings never provide actor attribution. Household profiles remain convenience selectors rather than access-control accounts. Legacy Nutrition credentials and permission mutation are fail-closed, and profile data is archived through the household lifecycle rather than destructively deleted.

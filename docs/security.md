# Security boundary

This app is designed for a trusted private household network, not the public internet. Profiles are explicitly not accounts, passwords, or authorization. The UI shows this warning during setup and after setup.

Current controls:

- SQLite, migrations, filesystem paths, and OpenAI credentials are server-only.
- POST/PATCH endpoints require an exact trusted `Origin`; CORS is not enabled by default.
- The active-profile selection is a signed, HttpOnly, `SameSite=Lax` cookie; its secure flag is enabled in production.
- Production refuses a missing or short `COOKIE_SECRET`.
- Baseline CSP, anti-framing, nosniff, referrer, permissions, COOP, and CORP headers are applied application-wide.
- Input is Zod-validated at the HTTP boundary. OpenAI provider modules are marked server-only; `OPENAI_API_KEY` takes precedence, while ignored `.api_keys` may be read only in local development and is excluded from Docker.
- AI readiness reports only configured/unconfigured state. OpenAI review and image requests require trusted origin, active profile, literal confirmation, bounded source input, a four-action-per-profile ten-minute process limit, strict Structured Outputs, and an audit row containing only source digest/label, operation/model/status/timestamps, and local IDs. Raw source, provider URLs, and credentials are neither audited nor returned. Tests use deterministic doubles; no automatic or live paid request is made here.
- Production dependency releases are installed only through frozen pnpm metadata. The
  workspace narrowly resolves Next's vulnerable declared PostCSS child to pinned
  `postcss@8.5.17`; `pnpm audit --prod --audit-level=moderate` is a release-maintenance
  gate and was clean on 2026-07-13. See decision 0024 before changing that override.
- Public URL capture is server-only and draft-only: it rejects credentials, private/reserved resolved addresses, excess redirects, non-text content, large responses, and slow responses. Redirect targets are rechecked; normal tests use mocked DNS/fetch responses only.
- URL Recipe extraction receives only the already bounded HTML response. A server-only parser traverses static markup without executing page scripts, loading images, or following metadata URLs; it considers embedded Schema.org JSON-LD Recipe nodes, then Microdata, and otherwise returns an explicitly warned Open Graph/text review fallback. Candidate state and page HTML are never persisted, and a browser must explicitly select a candidate before review.
- Recipe-photo upload accepts only JPEG, PNG, and WebP signatures below 10 MB. HEIC/HEIF is decoded in browser memory to JPEG before multipart construction; raw HEIC/HEIF still fails the server's byte gate. Sharp decodes accepted bytes with a pixel limit, rejects excessive dimensions and multi-frame images, honors orientation, strips source metadata, bounds output to 1,600 pixels, and writes a normalized WebP using an opaque generated key below `DATA_DIR/uploads`.
- Recipe-photo reads are recipe-scoped and always serve `image/webp` with `nosniff`; neither user file names nor user paths reach the filesystem.
- Document imports accept one PDF or one to four JPEG/PNG/WebP sources with a combined 15 MB limit after byte-signature validation. Browser HEIC/HEIF conversion happens before this boundary, and a bounded conversion-label map only records that the original was not uploaded. PDFs are parsed only from supplied bytes with local PDF.js, have a 12-page and 100,000-character limit, and reject malformed/password-protected input. Each scan is decoded with Sharp pixel/frame limits, orientation-normalized, metadata-stripped, and stored as an opaque local WebP artifact. A filename is reduced to a safe display label and never used as a path; artifact reads require an import-scoped opaque ID, add `nosniff`, are private/no-store, and force PDF download rather than inline execution.
- Import creation requires an active profile and trusted origin and is bounded to six starts per profile per ten-minute process window. Clear English scan sets may use one locally packaged Tesseract WASM worker at a time (two bounded waiters); it resolves only a validated package filesystem path with cache writes disabled, accepts only the existing normalized image buffers, has a 45-second deadline, and returns a manual-transcription fallback for blank/low-confidence/timeout/error output. It neither accepts a remote URL nor uses the default CDN model path, telemetry, an OpenAI credential, or provider request. Textless PDFs remain manual. Every source hash/name/order, extraction method/text, local model/version/confidence metadata, warning, and confirmation actor are retained as local provenance for review. The OCR assist is not handwriting validation.
- JSON-LD portability is paste-only and capped at 1 MB before parsing. The parser accepts only JSON root objects/arrays and bounded root/`@graph` Recipe discovery; it never treats text as a URL, opens files, unpacks archives, or traverses arbitrary object graphs. Candidate discovery and review drafts are request-local rather than persisted. Trusted-origin, active-profile confirmation validates the ordinary recipe schema before any write; the export is a deterministic local transform with `nosniff` and no household-private fields.
- Portable full-recipe export is server-generated output only: an active profile is required, an explicit `Origin` must be trusted, and the gzip response is `no-store` with `nosniff`. It accepts no bytes, file names, archive paths, or restore input. Private staging receives only generated JSON-LD and copied regular normalized WebP files; source links/symlinks, missing media, JSON-LD/media files over 10 MB, a manifest over 16 MB, more than 10,000 recipes or 50,000 distinct images, and more than 1 GB uncompressed output are rejected. The manifest exposes only stable archive-relative paths, lengths, and SHA-256 checksums; it excludes profile identity, private preferences, revisions, import/OCR data, storage keys, and configuration. Staging/archive files are removed after a completed, failed, or cancelled stream.
- Rich recipe metadata, equipment, and nutrition values are bounded shared inputs. Nutrition is displayed and exported only as entered; there is no nutrition estimator, provider credential, remote lookup, or outbound request. Per-profile ratings and notes are read/written only with the validated active-profile ID, use a composite profile/recipe key, and are excluded from shared revision snapshots plus JSON-LD and Markdown exports.
- Recipe snapshots are internal bounded JSON produced by the ordinary recipe validator. A restore route requires trusted origin, a valid active profile, a positive source revision, and the caller’s expected current revision; it validates the saved snapshot again and writes a new revision rather than replacing historical data. The library projection exposes only the selected profile’s favorite/rating values, not another profile’s preference row.
- Meal-plan writes accept only a valid local date, one of breakfast/lunch/dinner/snack, bounded servings/note, and either a current recipe ID or a bounded free-form title. Calendar export is a deterministic local `text/calendar` response with `nosniff`; it does not contact a calendar provider. Shopping rows and aisle names/order are bounded and require a trusted origin; deleting an aisle unassigns rows rather than deleting household food data.
- The PWA cache is constrained to successful same-origin `GET` responses for already-viewed recipe reads, local images, the manifest, and static assets. It never caches or replays POST/PUT/PATCH/DELETE requests, does not queue background sync, and returns a static offline page for uncached navigation rather than simulating a write.
- Backup export uses a SQLite snapshot plus only regular local media files and safe display metadata; it excludes environment files and secrets. Restore accepts only a currently listed server-created bundle, validates archive entry paths/types/size limits in isolated staging, checks every manifest checksum and SQLite integrity, requires exact `RESTORE` confirmation, and creates a safety backup before any data-root replacement.
- The Docker runner is non-root and keeps mutable state only below `/data`. `COOKIE_SECRET` and trusted origins are runtime environment variables, never image layers or backup exports. The checked-in container configuration remains LAN-oriented and must not be exposed publicly without a real authentication/reverse-proxy design.

Before adding arbitrary archive uploads, bundled handwriting OCR, remote-image imports, nutrition providers, or remote backup, add the corresponding size/type limits, audit/retention rules, mock-based tests, and any required human permission. A live OpenAI request additionally requires explicit paid-call approval.

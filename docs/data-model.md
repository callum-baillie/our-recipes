# Data model

## Foundation tables

| Table                                            | Purpose                                                                             | Key fields                                                                                                                                                            |
| ------------------------------------------------ | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `households`                                     | Exactly one local household/application setup record.                               | `id`, `name`, `app_name`, timestamps                                                                                                                                  |
| `profiles`                                       | Switchable personal preferences in the shared household. Not accounts.              | `id`, `display_name`, `color`, optional `avatar_url`, units, temperature unit, locale, timezone, `archived_at`, timestamps                                            |
| `recipes`                                        | Shared current recipe card, lifecycle, metadata, nutrition, and actor attribution.  | title, summary, status, yield, prep/cook/rest times, difficulty, cuisine, category, source/author/method, user-entered nutrition, tips, notes, actor IDs              |
| `recipe_ingredient_groups`, `recipe_ingredients` | Ordered structured ingredient content.                                              | group/item positions, optional quantity, unit, item, note                                                                                                             |
| `recipe_instruction_sections`, `recipe_steps`    | Ordered structured cooking method.                                                  | section/step positions and step body                                                                                                                                  |
| `recipe_equipment`                               | Ordered shared equipment names for a recipe card.                                   | recipe ID, position, bounded name                                                                                                                                     |
| `tags`, `recipe_tags`, `recipe_revisions`        | Household tag metadata/usage, recipe associations, and immutable revisions.         | normalized tag name/color; recipe association; revision number, JSON snapshot, editor, timestamp                                                                      |
| `recipe_images`                                  | Locally processed recipe media, never a client filesystem path.                     | opaque storage key, alt text, normalized dimensions, creator, timestamp                                                                                               |
| `collections`, `collection_recipes`              | Curated, manually ordered household cookbooks and ordered many-to-many membership.  | collection name/description/position, optional member-recipe image cover, creator/editor; recipe membership position and attribution                                  |
| `import_operations`, `import_artifacts`          | Review-only local PDF or ordered scan-set provenance and final confirmation link.   | operation extraction text/method/OCR model-confidence metadata/warnings/creator/confirmation; per-source opaque ID/key, safe label, SHA-256, media type, and position |
| `meal_plan_entries`                              | Dated breakfast/lunch/dinner/snack entries referencing a recipe or free-form title. | date, meal, nullable recipe, bounded title, planned servings, note, creator/editor                                                                                    |
| `shopping_aisles`                                | Household-defined store departments with a durable display order.                   | unique bounded name, position, timestamps                                                                                                                             |
| `shopping_lists`, `shopping_list_items`          | Generated list shell plus editable household item rows.                             | week range, ordered item, quantity/unit/note, nullable aisle, checked state, source recipe IDs                                                                        |
| `recipe_favorites`, `cook_sessions`              | Per-profile cooking preferences and completion history.                             | profile/recipe pair; target servings, started/completed timestamps                                                                                                    |
| `recipe_profile_preferences`                     | Private per-profile rating and kitchen note, separate from shared revisions.        | profile/recipe pair, nullable 1–5 rating, bounded note, updated timestamp                                                                                             |
| `ai_operation_audits`                            | Content-free audit trail for explicit AI actions.                                   | operation kind/status, source SHA-256 digest/label, provider/model, actor profile, optional recipe/import/generated-image IDs, timestamps                             |

Backup manifests are not application-table rows. They are versioned sidecar records inside server-created recovery bundles, so a corrupted or restored database cannot make an arbitrary archive appear trusted.

The initial migration is immutable and applied through Drizzle. IDs are server-generated UUIDs; timestamps are stored as SQLite integer timestamps through Drizzle’s timestamp mode.

## Planned ownership model

Recipes belong to the household; profile-specific fields such as favorites, ratings, personal notes, recent views, and defaults do not change shared recipe content. SQLite FTS5 indexes the current recipe title, summary, ingredients, tags, category, and cuisine; the application refreshes the index inside the create/edit transaction. Recipe graph writes require the caller's current revision and append a structured snapshot, so another tab cannot silently replace a newer edit. Restoring a revision parses that existing snapshot through the current schema and uses the same append-only update path; it never overwrites the source snapshot. The profile-preference primary key prevents one profile’s rating/note from being returned for another profile and its updates never append a recipe revision. Every import/capture stays reviewable before becoming shared content.

The `tags` table is the household’s canonical tag catalog; `recipe_tags` retains a compact normalized association for the FTS/search path. Tag rename, merge, and removal preserve or deliberately remove associations transactionally, then refresh affected FTS rows. Archived profiles remain in audit history but cannot become the active profile, and the service refuses to archive the last active profile.

Collections are separate from the recipe revision graph: one recipe may have an ordered membership in many collections without treating a shelf edit as a recipe-content revision. A collection cover is a nullable foreign key to an already-normalized `recipe_images` row belonging to one of that collection’s member recipes. Deleting the cover photo or removing its recipe clears the reference; deleting a collection cascades only its membership rows.

Recipe image rows retain only an opaque server-generated key and safe presentation metadata. The corresponding normalized WebP is below `DATA_DIR/uploads`; client filenames and MIME declarations are never used as a filesystem path or retained as canonical media metadata.

An import operation is not a recipe revision and does not itself publish shared recipe content. Its stored text is local PDF extraction, a user-provided combined scan transcription, or a local English scan-OCR suggestion used to produce the review draft. An `openai-vision-pending` scan row deliberately retains no source text: it establishes local artifact provenance before the user’s separate, credential-gated vision-review action; the provider candidate remains request-local. `local-ocr` rows additionally retain the immutable model ID, wrapper/data package versions, detected engine version, and rounded aggregate confidence; this is provenance, not a correctness claim. Every operation has ordered `import_artifacts`: a PDF has one artifact, and a scan set has one to four individually hashed normalized WebPs. The operation links to the exact recipe only after explicit confirmation. Imported PDFs and normalized scan WebPs use opaque keys below `DATA_DIR/generated/imports`, a subtree already included in local backup and restore.

Pasted Schema.org JSON-LD has no table, artifact, hash, or backup entry. Candidate discovery and draft mapping are request-local; only the user-confirmed ordinary recipe graph is persisted. JSON-LD export is likewise derived from the current recipe graph and does not add durable portability state.

Portable full-recipe export also adds no table or durable record. A request-local manifest derives its deterministic snapshot timestamp from the newest exported recipe update, maps positional JSON-LD documents to content-hash-addressed normalized WebP copies, and records only archive-relative paths, byte sizes, and checksums. It intentionally never records profile identity, source storage keys, personal preferences, revisions, import provenance, or archive paths in SQLite.

AI operation audits deliberately do not store pasted source, scan bytes, provider URLs, response text, or credentials. They establish that a particular profile explicitly requested an operation and whether the server stored a validated local result. A review candidate remains request-local and becomes recipe data only through the existing confirmation routes.

Public URL capture has the same no-retention model. Safely fetched bounded HTML, embedded structured metadata, candidate summaries, and Open Graph/text fallback data are request-local only. The confirmed recipe stores ordinary source attribution; no remote page body, image, cookie, or fetch credential is retained.

Generating a list never rewrites an existing list. It creates a new dated list and aggregates numeric structured quantities only when unit, item, and note match; source recipe IDs remain with each generated row for future explainability. Free-form meal entries deliberately contribute no derived ingredient. `shopping_aisles` is a household catalog, not a list copy: list rows retain a nullable aisle ID, rendered groups follow the saved aisle order, and aisle removal sets affected rows to unassigned rather than deleting them.

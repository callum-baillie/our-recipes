# Data model

## Foundation tables

| Table                                                      | Purpose                                                                                                          | Key fields                                                                                                                                                                 |
| ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `households`                                               | Exactly one local household/application setup record.                                                            | `id`, `name`, `app_name`, timestamps                                                                                                                                       |
| `profiles`                                                 | Switchable personal preferences in the shared household. Not accounts.                                           | `id`, `display_name`, `color`, optional `avatar_url`, units, temperature unit, locale, timezone, personal goal note and structured goal context, `archived_at`, timestamps |
| `recipes`                                                  | Shared current recipe card, lifecycle, metadata, nutrition, and actor attribution.                               | title, summary, status, yield, prep/cook/rest times, difficulty, cuisine, category, source/author/method, user-entered nutrition, tips, notes, actor IDs                   |
| `recipe_ingredient_groups`, `recipe_ingredients`           | Ordered structured ingredient content.                                                                           | group/item positions, optional quantity, unit, item, note                                                                                                                  |
| `recipe_instruction_sections`, `recipe_steps`              | Ordered structured cooking method.                                                                               | section/step positions and step body                                                                                                                                       |
| `recipe_equipment`                                         | Ordered shared equipment names for a recipe card.                                                                | recipe ID, position, bounded name                                                                                                                                          |
| `tags`, `recipe_tags`, `recipe_revisions`                  | Household tag metadata/usage, recipe associations, and immutable revisions.                                      | normalized tag name/color; recipe association; revision number, JSON snapshot, editor, timestamp                                                                           |
| `recipe_images`                                            | Locally processed recipe media, never a client filesystem path.                                                  | opaque storage key, alt text, normalized dimensions, creator, timestamp                                                                                                    |
| `collections`, `collection_recipes`                        | Curated, manually ordered household cookbooks and ordered many-to-many membership.                               | collection name/description/position, optional member-recipe image cover, creator/editor; recipe membership position and attribution                                       |
| `import_operations`, `import_artifacts`                    | Review-only local PDF or ordered scan-set provenance and final confirmation link.                                | operation extraction text/method/OCR model-confidence metadata/warnings/creator/confirmation; per-source opaque ID/key, safe label, SHA-256, media type, and position      |
| `meal_plan_entries`                                        | Dated breakfast/lunch/dinner/snack entries referencing a recipe or free-form title.                              | date, meal, nullable recipe, bounded title, planned servings, note, creator/editor                                                                                         |
| `shopping_aisles`                                          | Household-defined store departments with a durable display order.                                                | unique bounded name, position, timestamps                                                                                                                                  |
| `shopping_lists`, `shopping_list_items`                    | Generated list shell plus editable household item rows.                                                          | week range, ordered item, quantity/unit/note, nullable aisle, checked state, source recipe IDs                                                                             |
| `recipe_favorites`, `cook_sessions`                        | Per-profile cooking preferences and completion history.                                                          | profile/recipe pair; target servings, started/completed timestamps                                                                                                         |
| `recipe_profile_preferences`                               | Profile-scoped rating/reaction and kitchen note, separate from shared revisions; not an access-control boundary. | profile/recipe pair, nullable 1–5 rating (dislike 1, like 3, staple 5), bounded note, updated timestamp                                                                    |
| `pantry_products`, `pantry_product_aliases`                | Canonical household product catalog and searchable ingredient/name synonyms.                                     | normalized display name, brand/variant/category, units/storage defaults, staple targets, aliases, actor attribution, archive timestamp                                     |
| `pantry_product_identifiers`                               | Optional non-UI product identifiers reserved for imports/integrations.                                           | product ID, identifier kind/value, timestamps                                                                                                                              |
| `pantry_locations`                                         | Nested household storage tree independent of products and batches.                                               | parent ID, normalized sibling name, storage type, position, actor attribution, archive timestamp                                                                           |
| `pantry_batches`                                           | Physical stock lots for canonical products.                                                                      | exact or approximate quantity, unit/package fields, location, printed/derived dates, status, provenance links, version, actor attribution                                  |
| `pantry_inventory_events`                                  | Immutable actor-attributed stock history and reversible state transitions.                                       | batch/product, event kind, quantity delta/unit, previous/new state JSON, reason/note, related recipe/meal/list IDs, undo links, timestamp                                  |
| `recipe_ingredient_product_mappings`                       | Optional seam between a structured recipe ingredient row and a canonical Pantry product.                         | ingredient/product pair, manual/exact/alias/suggested match kind, compatible-variant and optional flags, actor attribution, timestamps                                     |
| `nutrient_definitions`                                     | Canonical nutrient registry and display semantics.                                                               | stable code, names/aliases, category, canonical unit, precision, goal semantic, dashboard order                                                                            |
| `nutrition_data_sources`                                   | Explicit provenance for legacy, manual, provider, laboratory, calculated, and reference data.                    | source type/name/provider/version, URL/citation/license, retrieval time, priority, metadata                                                                                |
| `food_nutrition_records`, `food_nutrient_values`           | Append-only nutrient fact revisions for a canonical Pantry product.                                              | product/revision/source, measurement basis and conversion evidence, confidence/completeness, superseded revision, sparse nutrient values                                   |
| `nutrition_calculation_versions`                           | Immutable identity for deterministic calculation algorithms and factor sets.                                     | algorithm/version, energy and retention factor versions, implementation digest, metadata                                                                                   |
| `recipe_nutrition_calculations`                            | Append-only recipe-revision calculations, including legacy snapshots.                                            | recipe/revision, calculation/source IDs, source digest, serving/final-weight evidence, confidence/completeness, superseded calculation                                     |
| `recipe_nutrition_contributions`, `recipe_nutrient_values` | Ingredient evidence trace and sparse total nutrient vector for one calculation.                                  | ingredient/product-record links, multipliers/yields/retention, optional/missing evidence, per-nutrient amount/confidence/completeness                                      |
| `ai_operation_audits`                                      | Content-free audit trail for explicit AI actions.                                                                | operation kind/status, source SHA-256 digest/label, provider/model, actor profile, optional recipe/import/generated-image IDs, timestamps                                  |

Backup manifests are not application-table rows. They are versioned sidecar records inside server-created recovery bundles, so a corrupted or restored database cannot make an arbitrary archive appear trusted.

The initial migration is immutable and applied through Drizzle. IDs are server-generated UUIDs; timestamps are stored as SQLite integer timestamps through Drizzle’s timestamp mode.

## Planned ownership model

Recipes belong to the household; profile-specific fields such as favorites, ratings, personal notes, recent views, and defaults do not change shared recipe content. SQLite FTS5 indexes the current recipe title, summary, ingredients, tags, category, and cuisine; the application refreshes the index inside the create/edit transaction. Recipe graph writes require the caller's current revision and append a structured snapshot, so another tab cannot silently replace a newer edit. Restoring a revision parses that existing snapshot through the current schema and uses the same append-only update path; it never overwrites the source snapshot. The profile-preference primary key prevents one profile’s rating/note from being returned for another profile and its updates never append a recipe revision. Every import/capture stays reviewable before becoming shared content.

The `tags` table is the household’s canonical tag catalog; `recipe_tags` retains a compact normalized association for the FTS/search path. Tag rename, merge, and removal preserve or deliberately remove associations transactionally, then refresh affected FTS rows. Archived profiles remain in audit history but cannot become the active profile, and the service refuses to archive the last active profile.

Collections are separate from the recipe revision graph: one recipe may have an ordered membership in many collections without treating a shelf edit as a recipe-content revision. A collection cover is a nullable foreign key to an already-normalized `recipe_images` row belonging to one of that collection’s member recipes. Deleting the cover photo or removing its recipe clears the reference; deleting a collection cascades only its membership rows.

Recipe image rows retain only an opaque server-generated key and safe presentation metadata. The corresponding normalized WebP is below `DATA_DIR/uploads`; client filenames and MIME declarations are never used as a filesystem path or retained as canonical media metadata.

An import operation is not a recipe revision and does not itself publish shared recipe content. Its stored text is local PDF extraction, a user-provided combined scan transcription, or a local English scan-OCR suggestion used to produce the review draft. An `openai-vision-pending` scan row deliberately retains no source text: it establishes local artifact provenance before the user’s separate, credential-gated vision-review action; the provider candidate remains request-local. `local-ocr` rows additionally retain the immutable model ID, wrapper/data package versions, detected engine version, and rounded aggregate confidence; this is provenance, not a correctness claim. Every operation has ordered `import_artifacts`: a PDF has one artifact, and a scan set has one to four individually hashed normalized WebPs. The operation links to the exact recipe only after explicit confirmation. Imported PDFs and normalized scan WebPs use opaque keys below `DATA_DIR/generated/imports`, a subtree already included in local backup and restore.

Pasted Schema.org JSON-LD has no table, artifact, hash, or backup entry. Candidate discovery and draft mapping are request-local; only the user-confirmed ordinary recipe graph is persisted. JSON-LD export is likewise derived from the current recipe graph and does not add durable portability state.

Portable full-recipe export also adds no table or durable record. A request-local manifest derives its deterministic snapshot timestamp from the newest exported recipe update, maps positional JSON-LD documents to content-hash-addressed normalized WebP copies, and records only archive-relative paths, byte sizes, and checksums. It intentionally never records profile identity, source storage keys, personal preferences, revisions, import provenance, or archive paths in SQLite.

AI operation audits deliberately do not store pasted source, scan bytes, provider URLs, response text, or credentials. They establish that a particular profile explicitly requested an operation and whether the server stored a validated local result. A review candidate remains request-local and becomes recipe data only through the existing confirmation routes.

Public URL capture has the same no-retention model. Safely fetched bounded HTML, embedded structured metadata, candidate summaries, and Open Graph/text fallback data are request-local only. The confirmed recipe stores ordinary source attribution; no remote page body, image, cookie, or fetch credential is retained.

Generating a list never rewrites an existing list. It creates a new dated list and aggregates numeric structured quantities only when unit, item, and note match; source recipe IDs remain with each generated row for future explainability. Free-form meal entries deliberately contribute no derived ingredient. `shopping_aisles` is a household catalog, not a list copy: list rows retain a nullable aisle ID, rendered groups follow the saved aisle order, and aisle removal sets affected rows to unassigned rather than deleting them.

Pantry separates a canonical product from each physical batch so two packages of the same item can retain different quantities, locations, purchase dates, expiry dates, prices, and states. A batch stores either an exact quantity with a registered unit or an explicit approximate state. Unit conversion is allowed only inside count, mass, or volume dimensions; arbitrary density conversions are not inferred. Product aliases belong to the catalog, while optional recipe-ingredient mappings preserve the original recipe wording and can later power availability and demand calculations without rewriting recipe revisions.

Availability and projected demand are derived views, not stored allocations. A manual mapping is actor-attributed on the mapping row while the original recipe ingredient and revision graph remain untouched. Serving-scaled requirements are exact only when the recipe yield, ingredient quantity, unit, and compatible exact Pantry batch quantities are all known. Approximate batches, incompatible units, missing mappings, and missing quantities remain explicit unknowns. Weekly demand groups compatible requirements across every recipe-linked meal and counts each physical exact batch at most once; reading demand never changes or reserves a batch.

All Pantry mutations happen server-side and append `pantry_inventory_events` in the same SQLite transaction as the version-protected batch update. Event rows retain previous and new mutable state plus actor attribution and optional recipe, meal-plan, and shopping-list provenance. Undo is an append-only compensating event linked to the reversed event. Default household locations are seeded only when the first selected profile opens Pantry; archived locations remain referentially stable, and a location containing active stock or live child locations cannot be archived.

Split creates a second physical batch while reducing the source; combine depletes the selected peer while increasing the destination. Their two events carry a bounded pair reference, and each resulting batch version must still be current for paired undo. No new table is required: the existing batch version, before/after snapshots, reversal link, and event note preserve the reversible audit trail.

Migration `0020` adds `pantry_shopping_item_details` for formula snapshots, complete per-meal recipe/date/serving/contribution provenance, generation identity, null-preserving uncertainty, and field-level override flags. The companion shopping row stores the contributing recipe IDs. If demand becomes obsolete, a retained edited row clears those recipe IDs and all generated formula/provenance, uncertainty, generated-unit, and numeric fields while preserving its editable values and override flags. `pantry_purchase_intakes` gives a shopping item and explicit operation key one location-linked batch; rejected, unknown-outcome, and concurrent retries keep that key, while confirmed success rotates the browser tracker for a later distinct intake. Cook sessions may reference a planned meal. `pantry_cook_session_plans`, `pantry_cook_session_deductions`, and `pantry_cook_session_leftovers` retain the confirmation snapshot and exact event/batch links. Inventory events gain a nullable cook-session link. Compensating events and the cook-plan `undone` state commit together. These records are additive; applied migrations and recipe ingredient text remain unchanged.

Migration `0026_pantry_grocery_formula` adds only grocery-generation mode, durable coverage/control state, manual extra, covered and purchased quantities/units, and a bounded control note. A zero-shortage row remains stored when a missing-only view hides it, so covered state and later all-item regeneration remain stable. Rich intake continues to use the existing Pantry batch fields rather than duplicating package, location, date, price, store/source, or notes columns.

Nutrition facts do not live on a Pantry batch. `food_nutrition_records` attaches a sourced revision to the canonical product, while its sparse value rows distinguish missing data from a measured zero. A later revision must name the latest record it supersedes; existing record/value rows are never edited by the service. Source priority is selection policy, not deletion or truth by assertion. Cross-dimension conversion requires stored density or piece/serving-weight evidence.

Recipe nutrition follows the same history rule. Each calculation names the exact recipe revision, calculation version, source digest, source record, completeness, and confidence. Contribution rows retain mapped product evidence or an explicit missing reason plus yield/retention inputs. Migration `0016` creates one immutable calculation for recipes with any of the eight legacy fields and marks that snapshot partial and non-authoritative. The old columns remain during transition. Future intake rows must copy nutrient/source/calculation snapshots rather than point at mutable preferred records.

`nutrition_profiles.linked_household_profile_id` is a required, unique one-to-one link after migration `0028`; `owner_principal_id` is also unique and identifies the compatibility principal provisioned for that household profile. The principal ID is deterministic and collision-checked, credentials are replaced by a non-authenticating sentinel, and permission history cannot be extended. Nutrition goal, measurement, intake, allocation, prepared-instance, consumption-command, diary-command, and recommendation-feedback history rows retain both `created_by_principal_id` and nullable `actor_household_profile_id`; new writes require the exact active pair.

Migration `0025` adds only `meal_plan_entries.status`, constrained by application/schema typing to `planned`, `skipped`, or `cancelled`, plus an index for projection reads. `cooked` is deliberately not stored: it is derived from a completed cook session linked to the meal-plan entry. Pantry demand ignores skipped, cancelled, and derived-cooked entries; changing those statuses never mutates a Pantry batch.

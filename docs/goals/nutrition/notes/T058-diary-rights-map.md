# T058 Food Diary lifecycle and data-rights map

The safe implementation splits into two sequential Workers.

Worker A adds an append-only `nutrition_diary_commands` table keyed by principal and bounded idempotency key with canonical request digest and result metadata. Server-owned copy-entry/day reads latest consumed snapshots and creates new immutable series/times; move appends a corrected revision with the same frozen snapshot; restore finds the last consumed snapshot before a deleted latest revision and appends an audited correction; reassignment authorizes manage access to both profiles and atomically creates target history plus a source deletion revision. Matching retries return stored results; conflicting key reuse fails. A deterministic profile export uses `export_data`, bounded JSON, no-store/nosniff attachment headers, and includes only the authorized profile, current/history records, goals, measurements, permissions, allocations and prepared links needed to interpret them—never credentials, hashes, sessions, secrets or other profiles.

Worker B is destructive and must follow separately. It requires `delete_data`, exact `DELETE <display name>` confirmation and expected profile version. Dependency order is: diary/consumption command rows; intake nutrient children; newest-to-oldest intake and allocation self-referential versions; newest-to-oldest goal and permission versions; measurements; profile-linked remaining private rows; then scrub sensitive profile fields, archive and increment version. Prepared instances are principal-owned operational snapshots, so delete only those proven exclusive and unreferenced; otherwise detach/scrub their private notes. Archive/rotate the principal only when no other active owned/managed profile or referenced Nutrition record needs it. Never delete household recipes, calculations, references, Pantry or other profiles.

The exact GoalBuddy Scout exceeded the single-wait limit and was interrupted. The PM completed the read-only dependency map as permitted fallback.
